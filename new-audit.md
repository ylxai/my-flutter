## CRITICAL ISSUES (Must Fix Before Release)

### 1. Panic Propagation Across FFI Boundary
- **Location:** All functions in `rust/src/api.rs:99-359`
- **Detected language:** Rust
- **Issue:** No `std::panic::catch_unwind` wrapper. Any panic (e.g., from `unwrap()`, array bounds, allocation failure) will cross FFI boundary causing undefined behavior and application crash.
- **Native error path:** Any internal panic
- **Dart propagation:** None - application crash
- **Severity:** CRITICAL
- **Status:** FIXED
- **Resolution:** Semua fungsi API publik sudah dibungkus `std::panic::catch_unwind` dan dialihkan ke fungsi `_inner` agar panic tidak melewati batas FFI. Fallback aman disediakan saat panic terjadi.
- **Fix (implemented):**
```rust
pub fn copy_single_file(
    source: String,
    destination: String,
    skip_existing: bool,
) -> NativeFileCopyResult {
    std::panic::catch_unwind(std::panic::AssertUnwindSafe(|| {
        copy_single_file_inner(source, destination, skip_existing)
    }))
    .unwrap_or_else(|_| NativeFileCopyResult {
        source_path: source,
        dest_path: destination,
        bytes_copied: 0,
        duration_ms: 0,
        speed_mbps: 0.0,
        strategy_used: "Panic".to_string(),
        success: false,
        error_message: "Internal panic - operation failed".to_string(),
        skipped: false,
    })
}
```

### 2. Race Condition on Global Pause/Cancel Flags
- **Location:** `rust/src/api.rs:91-94` and `rust/src/api.rs:142-144, 193-205`
- **Detected language:** Rust
- **Issue:** Static `CANCEL_FLAG` and `PAUSE_FLAG` use `Ordering::Relaxed`. Concurrent calls to `copy_files_batch` from different Dart isolates or rapid successive calls will corrupt state. Line 142-143 resets flags tanpa sinkronisasi dengan operasi yang sedang berjalan.
- **Crash or race condition risk:** HIGH - Data corruption, stuck operations, incorrect cancellation, operations that cannot be cancelled
- **Severity:** CRITICAL
- **Status:** FIXED
- **Resolution:** Mengganti flag global menjadi token per operasi lewat `CURRENT_OPERATION` dan `OperationFlags`. Semua operasi pause/cancel menggunakan `Ordering::SeqCst` dan hanya berlaku untuk batch aktif.
- **Fix (implemented):**
```rust
#[derive(Debug)]
struct OperationFlags {
    cancel: Arc<AtomicBool>,
    pause: Arc<AtomicBool>,
}

static CURRENT_OPERATION: std::sync::LazyLock<Mutex<Option<Arc<OperationFlags>>>> =
    std::sync::LazyLock::new(|| Mutex::new(None));

fn copy_files_batch_inner(...) -> NativeBatchResult {
    let op_flags = Arc::new(OperationFlags {
        cancel: Arc::new(AtomicBool::new(false)),
        pause: Arc::new(AtomicBool::new(false)),
    });

    {
        let mut current = CURRENT_OPERATION
            .lock()
            .expect("CURRENT_OPERATION mutex poisoned");
        if let Some(existing) = current.as_ref() {
            existing.cancel.store(true, Ordering::SeqCst);
        }
        *current = Some(Arc::clone(&op_flags));
    }

    // ... gunakan op_flags.cancel & op_flags.pause
}

pub fn cancel_copy() {
    if let Some(current) = CURRENT_OPERATION.lock().expect("CURRENT_OPERATION mutex poisoned").as_ref() {
        current.cancel.store(true, Ordering::SeqCst);
    }
}
```

### 7. Busy-Wait Pause Implementation
- **Location:** `rust/src/parallel.rs:96-101`
- **Detected language:** Rust
- **Issue:** Pause uses spin-loop with 100ms sleep. Wastes CPU and can miss cancel signal.
- **Crash or race condition risk:** LOW - Performance/CPU waste
- **Severity:** MEDIUM
- **Status:** FIXED
- **Resolution:** Pause kini menggunakan `Condvar` lewat `PauseController`, sehingga thread menunggu dengan efisien dan segera dibangunkan saat resume/cancel.
- **Fix (implemented):**
```rust
use std::sync::{Arc, Condvar, Mutex};

#[derive(Debug)]
struct PauseController {
    paused: Mutex<bool>,
    condvar: Condvar,
}

impl PauseController {
    fn wait_while_paused(&self, cancel_flag: &AtomicBool) {
        let mut paused = self.paused.lock().unwrap();
        while *paused && !cancel_flag.load(Ordering::SeqCst) {
            paused = self.condvar.wait(paused).unwrap();
        }
    }
}
    }
    
    pub fn set_paused(&self, value: bool) {
        let mut paused = self.paused.lock().unwrap();
        *paused = value;
        if !value {
            self.condvar.notify_all();
        }
    }
}
```

- **Location:** `rust/src/platform/windows.rs:11-20`
- **Detected language:** Rust
- **Issue:** Windows implementation just calls `fs::copy` - no mmap or platform-specific optimization
- **Evidence:** `fs::copy(src, dst)` - standard library copy
- **Severity:** LOW (Performance)
- **Status:** FIXED

```rust
pub fn copy_file_win32(src: &Path, dst: &Path) -> io::Result<u64> {
    let src_wide: Vec<u16> = OsStr::new(src)
        .encode_wide()
        .chain(Some(0))
        .collect();
    let dst_wide: Vec<u16> = OsStr::new(dst)
        .encode_wide()
        .chain(Some(0))
        .collect();

    unsafe {
        let success = CopyFileExW(
            PCWSTR(src_wide.as_ptr()),
            PCWSTR(dst_wide.as_ptr()),
            None,
            None,
            None,
            windows::Win32::Storage::FileSystem::COPY_FILE_FLAGS(0),
        );

        if !success.as_bool() {
            return Err(io::Error::last_os_error());
        }
    }

    fs::metadata(dst).map(|meta| meta.len())
}
```

## PRODUCTION VERDICT

### First Likely Native Failure:
**Panic during memory-mapped copy** - If mmap fails unexpectedly or an allocation panics, the application will crash with undefined behavior. No recovery possible.

### Must-Fix Before Release:
1. Add `std::panic::catch_unwind` wrapper around all FFI-exposed functions
2. Replace `Ordering::Relaxed` with `Ordering::SeqCst` or use proper mutex for pause/cancel state
3. Validate and canonicalize file paths before operations
4. Fix race condition on global state

### Acceptable Risks:
- Missing `#[repr(C)]` - serialization layer protects ABI
- Busy-wait pause - functional, just inefficient
- Per-call ThreadPool creation - acceptable for typical batch sizes
- Windows lack of mmap optimization - functional, just slower
- Low-severity performance issues
